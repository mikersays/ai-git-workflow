P. Prologue A mental model

What git actually is

Forget every intimidating thing you've heard. Git is a save system for code. That's it. When you're playing a game and you reach a boss fight, you save first. When Claude has just helped you get the login screen working, you save the project. Same idea.

The complication is only that there are three places code can live, and once you can picture them, the rest of git becomes obvious. Here they are:

Place 01 · local

Your folder

The files on your laptop right now. The thing Claude is editing. This is where you build.

→
Place 02 · history

Your local repo

A hidden journal inside your folder (the .git directory) that remembers every save you've ever made.

→
Place 03 · remote

GitHub

A copy of that journal living on a server. Your backup. The thing other machines can pull from.

When you "save" with git, you're really doing two motions: commit (write a new entry into the local journal) and push (upload that entry to GitHub). The /ship:ship command does both in one breath so you don't have to think about it.

Below is the rest of the git vocabulary you'll meet, translated into things you've already used a hundred times — game saves, Word's "Save As," Google Drive. Skim it once; come back when a word in a later stage feels foreign.

The git word
What it actually means
If you've used a computer, it's like…
commit
a single save in your project's journal
a save point in a video game — name it, return to it later
push
upload your saves to GitHub
syncing a Word doc up to Google Drive
pull
download saves you don't have yet
opening the latest Drive copy of a shared doc
repo
your project folder + every save you've ever made
a Word doc with full version history attached

"Save As: experiment.docx" — the original is untouched

Save early. Save often. You can always go back to a save. You can rarely undo what you didn't. Rule №1

That single instinct — "I just got something working, save it" — is 80% of git as a habit. The remaining stages teach you the buttons. Take them in order the first time, then return only to the parts you need.

00 Setup ~5 minutes

Install the three tools

You need three things on your machine. If you already have one, skip its line. Run these in your terminal — that's the black-and-white window where you type commands. On Mac, open Terminal. On Windows, install WSL and use the Ubuntu prompt. On Linux, you already know.

Git itself

The thing all of this is built on. Most systems already have it.

terminal
$ git --version
git version 2.43.0   # anything 2.x is fine

If that errored: on Mac run xcode-select --install; on Ubuntu/Debian run sudo apt install git; on Windows install Git for Windows.

The GitHub CLI (gh)

gh is GitHub's official command-line tool. It's the easiest way to log in once and then create repositories without ever touching the website.

terminal · install gh
# macOS
$ brew install gh

# Ubuntu / Debian / WSL
$ sudo apt install gh

# anywhere else: see https://cli.github.com

# confirm it works
$ gh --version
gh version 2.x.x

Claude Code

The CLI you'll do most of your building in. It runs in your terminal and edits files, runs commands, and ships work — when you let it.

terminal · install claude
$ npm install -g @anthropic-ai/claude-code

# first run will prompt you to log in
$ claude

If npm isn't on your machine yet, install Node.js from nodejs.org first. Pick the version labeled LTS.

02
02 Repo ~3 minutes

Make your first repository

A "repository" — repo, for short — is the project folder, plus its hidden journal of saves. You'll have one repo per project. There are two ways to start, depending on whether you already have a folder or not.

Path A — starting fresh

You don't have any code yet — you want to begin a brand-new project.

terminal · path A
# create the repo on GitHub AND on your laptop, in one step
$ gh repo create my-cool-app --private --clone

# step into it
$ cd my-cool-app

# now start Claude here and start building
$ claude

The flag --private keeps the project to yourself. Use --public if you want anyone to be able to see it. --clone is what creates the local folder.

Path B — you already have a folder

You already started a project — maybe Claude has been editing files in some folder for an hour and you're only now hearing about git. This is the most common case. We're going to adopt your existing folder into git in four small steps.

terminal · path B
# step into your existing folder
$ cd path/to/your/project

# 1. start the local journal (creates the hidden .git folder)
$ git init
Initialized empty Git repository in .../.git/

# 2. stage every file in the folder
$ git add .

# 3. make the very first save
$ git commit -m "first commit"

# 4. create a matching repo on GitHub and push everything up
$ gh repo create --source=. --private --push

✓ Created repository your-username/your-project on GitHub
✓ Pushed commits to https://github.com/your-username/your-project

Four small commands. The first three turn your folder into a git repo and bookmark its current state. The fourth uploads it to GitHub. After this you'll never need to do any of them again on this project — from here on, /ship:ship handles everything.

Path A created an empty repo on GitHub and copied it down to your laptop — you start with nothing. Path B took the folder you already had, started a journal in it, made the first save, created a matching repo on GitHub, and uploaded everything — you start with whatever you'd built. Either way, you now have a project that exists in all three places from the prologue.

About .gitignore — what not to save

Some files don't belong in git. Downloaded packages (the node_modules folder), build output, OS junk like .DS_Store, and most importantly any file that contains an API key, password, or other secret. Without a .gitignore, git will happily save them all and push them to GitHub for the world to see.

A .gitignore is a plain text file at the root of your project. Each line is a pattern of files to skip. Make one as soon as your project exists, ideally before your first commit.

.gitignore · example for a Node.js project
# downloaded packages — never commit these
node_modules/

# environment files with secrets
.env
.env.local
.env.*.local

# build output
dist/
build/
.next/
.cache/

# OS & editor junk
.DS_Store
Thumbs.db
.vscode/
.idea/

# logs
*.log
npm-debug.log*

Different stacks (Python, Rust, Next.js, Swift…) need different patterns. You don't have to memorize them. The fastest path is to ask Claude to read your project once and write the right one for you.

If you committed a real secret (API key, password) by mistake — rotate the secret immediately on whatever service issued it. Then add it to .gitignore and ask Claude to help untrack the file. Removing a secret from git history is harder than just rotating it, so save your future self the trip.

When to actually run /ship:ship

  • Before you try something risky. About to ask Claude to refactor? Ship the working state first.
  • End of every session. Closing the laptop = ship first. Future-you will thank you.

When not to ship

  • The change is half-finished and broken — finish or revert first.
  • You just added a real password or API key to a file by accident — remove it before shipping.

06
06 Branch Risky moves

Branches, for when you want to try something

A branch is a parallel save file. Your project keeps a main timeline (called main), and you can spin off a side timeline whenever you want to experiment without affecting the working version. Think of it like clicking "Save As: experiment.docx" in Word — the original document is untouched while you mess around with the copy. If the experiment works, you copy the changes back. If it doesn't, you throw the experiment away and your original is still pristine.

When to branch

  • Big rewrites or refactors that might leave the app broken for a while.
  • "What if we changed the whole color scheme" experiments.
  • A new feature that will take a few sessions to finish.
  • Anything you want a clean rollback from.

Where does /ship:ship send my work?

Whatever branch you're currently on. That's it. Shipping is "save and push this branch," not "send to main." This is the whole reason branches are safe — main only changes when you decide it does.

  • Shipping while on main — /ship:ship updates main on GitHub. This is your usual day-to-day.
  • Shipping while on a side branch — /ship:ship updates only that branch on GitHub. Main stays untouched until you merge.

If you ever wonder where you are, ask:

terminal · which branch?
$ git branch --show-current
main                          # shipping goes to main

# or, after creating a side branch:
$ git branch --show-current
try-new-onboarding           # shipping goes only to this branch

The takeaway: the switch is the decision; the ship just executes. Want this change on main? Switch to main first, then ship. Want it isolated? Switch to a branch first, then ship.

Make a branch and switch to it

terminal · branching
# confirm where you are
$ git status
On branch main
nothing to commit, working tree clean

# create a new branch and jump onto it
$ git switch -c try-new-onboarding
Switched to a new branch 'try-new-onboarding'

# now build with claude as usual, ship as usual
# /ship:ship will push the branch, not main

If the experiment worked → merge it

If you like the result, you merge the side timeline back into main. The cleanest way is on github.com via a pull request, which gives you a chance to look at every change before accepting it.

terminal · pull request
# open a pull request from your branch into main
$ gh pr create --fill

# after reviewing it on github.com, merge it
$ gh pr merge --squash --delete-branch

# return to main and grab the merged version
$ git switch main
$ git pull

If the experiment failed → walk away

This is the magic. You don't have to undo anything. Just go back to main. The branch still exists in case you want to revisit it later, but main is untouched.

terminal · abandon
$ git switch main
Switched to branch 'main'

# your files are now back to the way main looks. nothing was lost.
# the side branch is still on github if you ever want it.

Branches are the fearlessness pattern. You can always come back. Rule №4

Naming branches

Lowercase, hyphens, descriptive: add-login, try-new-colors, fix-mobile-nav.

Don't pre-branch

You don't need a branch for every change. Reach for one when the change is risky, big, or unfinished. Small confident edits go straight to main.

Switch ≠ checkout

Older guides use git checkout. git switch is the modern, less-confusing version. Use that.

07
07 Recover The safety net

Time travel & undo

Every commit you've ever made is a save point you can return to. This stage covers the most common five flavors of "I want to go back" — from gentle to nuclear.

Just look at history

See your last several saves with their messages.

terminal · history
$ git log --oneline -10
def4567 add password reset flow
abc1234 style the login button
9876fed first working signup
...

Throw away unsaved changes to a single file

You changed a file, regret it, and haven't shipped yet. Restore it to the last save.

terminal · restore one file
$ git restore path/to/file.tsx

Throw away all unsaved changes

The whole working state since the last commit, gone. Useful when an AI session went sideways.

terminal · reset to last save
$ git restore .

# or, if you also created new files you want gone:
$ git clean -fd

Roll the project back to an older save

You've shipped a few things, and a recent one made everything worse. You want to return to a commit from earlier. Use git revert, which adds a new save that undoes a previous one — non-destructive.

terminal · revert
# find the bad commit's hash from git log
$ git log --oneline -10
def4567 broke everything            # ← this one
abc1234 style the login button

# undo it with a new save (the safe way)
$ git revert def4567
$ git push

git revert creates a new save that undoes the bad one — your history grows forward, never disappears. git reset --hard deletes saves. For beginners, always revert. You will keep your sanity and your team (or future-you) will keep their work.

Just ask Claude

The best safety net is the assistant sitting next to you. If something is broken and you don't know which lever to pull, paste the error or the situation into Claude and say "I need to get back to a working state without losing too much. What are my options?" Claude can read your git status and your git log and pick the right tool — usually one of the four above.

The work isn't really gone unless you typed --force. Rule №5

Avoid by default

Until you're comfortable: never run git push --force, git reset --hard, or anything ending in -f. They are the rare commands that can destroy your work.

Your strongest move

"Show me git status and git log --oneline -10, then suggest the safest path." — paste this into Claude any time you're stuck.

Also good

GitHub keeps everything you've pushed. Even a deleted local folder is recoverable if you'd shipped it.

Field notes

Things that seem wrong but aren't

№01 I shipped, then realized the change was bad. Did I lose the old version? +

No. Every previous save is still in git log. Use git revert <commit> (Stage 07) to add a new save that undoes the bad one. Your history grows forward.

№02 Should I learn the deeper git commands eventually? +

Eventually, sure. There's an entire universe of rebase, cherry-pick, stash, bisect, and so on. You don't need them to ship. Most professionals get by with what's in this guide plus two or three more commands they looked up the day they needed them. Build the habit first; learn the side-quests when one of them blocks you.

№03 My project has secrets in a file. Can I still push? +

Move the secrets out first. Create a file called .env that contains your secrets, then create a file called .gitignore at the root of your project with a single line: .env. That tells git to never include that file in saves. Then ship.

If you already pushed a secret by accident: rotate the secret immediately (i.e., change it on whatever service issued it). Removing it from history is harder than rotating, so save your future self the time and just rotate.

№04 Why do I keep seeing "main" — I thought it was "master"? +

GitHub renamed the default branch from master to main a few years back. Modern projects use main. Older ones still use master. They mean the same thing. The git config --global init.defaultBranch main line in Stage 01 makes sure all your new projects start with main.

№05 What's the difference between git pull and git fetch? +

git fetch downloads new saves from GitHub but doesn't apply them to your files. git pull downloads and applies. For everything in this guide, you want git pull. Forget fetch until you have a reason to remember it.

№06 Claude added 200 files I didn't expect. Should I ship them? +

Probably they're node_modules/, build artifacts, or caches that don't belong in git. Add a .gitignore with the right patterns first — most frameworks have a standard one (try github/gitignore). Or just ask: "Generate a sensible .gitignore for this project." Claude will read your stack and make the right one.

№07 I'm scared. What if I run the wrong command and break everything? +

The everyday commands in this guide cannot lose work that you've shipped. The dangerous ones all involve the words --force, --hard, or -f. Avoid those for now and you have a near-perfect safety net.

When in doubt, run git status. It tells you exactly where you are and what would change. It cannot do harm. Run it as often as you want.

№08 Do I really need branches if I'm working alone? +

Not for everyday work. Ship to main as you go. Reach for a branch only when you want a clean rollback for something experimental. Solo developers using AI assistants often go weeks at a time without making a branch, and that's fine.
